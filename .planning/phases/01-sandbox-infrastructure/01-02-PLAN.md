```typescript
import Docker from "dockerode";
import { nanoid } from "nanoid";
import type { SandboxConfig, ContainerInfo } from "@stallion/shared";

const CONTAINER_IMAGE = "stallion-agent-control:latest";
const CONTROL_PORT_INTERNAL = 3001;

export class ContainerManager {
  private docker: Docker;

  constructor(dockerSocket = "/var/run/docker.sock") {
    this.docker = new Docker({ socketPath: dockerSocket });
  }

  async createSessionContainer(config: SandboxConfig, proxyPort: number): Promise<ContainerInfo> { ... }
  async destroySessionContainer(containerId: string): Promise<void> { ... }
  async sweepOrphans(): Promise<number> { ... }
  async waitForReady(hostPort: number, authToken: string, maxWaitMs?: number): Promise<void> { ... }
  async copyWorkspaceFromContainer(containerId: string, containerPath: string, hostPath: string): Promise<void> { ... }
}
```

Key implementation details:
- `createSessionContainer`: Generate auth token via `nanoid(32)`. Use `HostPort: "0"` to let Docker assign a free port (avoids Pitfall 6 from research). After start, inspect container to read assigned port from `NetworkSettings.Ports["3001/tcp"][0].HostPort`.
- `waitForReady`: Exponential backoff polling GET `http://localhost:<hostPort>/status` with `x-control-token` header. Use `fetch()` (Node 22 has native fetch). Throw on timeout (Pitfall 1 from research).
- `sweepOrphans`: `docker.listContainers({ all: true, filters: JSON.stringify({ label: ["stallion.managed=true"] }) })`, then remove each with `force: true`.
- `copyWorkspaceFromContainer`: Use `docker cp` equivalent via `container.getArchive({ path: containerPath })`. Extract tar to host path.

**Refactor MissionManager to create Docker containers instead of MissionEngine instances.**

This is the critical integration point. The existing `MissionManager` currently does:
1. `new MissionEngine(envConfig)` on startMission
2. `engine.execute(prompt, id, workspace, onSDKMessage, onLifecycle)` to run the session
3. Direct workspace access via `walkDir()`

After refactor, it will:
1. Use `ContainerManager.createSessionContainer()` to create a Docker container
2. Use `ContainerManager.waitForReady()` to wait for the control server
3. Use `ContainerClient.startSession()` to send the prompt to the container
4. Use `ContainerClient.connectEvents()` to subscribe to SDK events via WS
5. Relay received events through the existing `notifySDK()` / `notifyEvent()` pipeline (zero frontend changes)
6. On session end: copy JSONL via SessionStore, copy workspace files, destroy container

**Changes to MissionData interface:**

**Changes to MissionSnapshot (for persistence):**
Add `containerId`, `hostPort`, `authToken`, `sdkSessionId` to the snapshot type so they survive backend restarts. Do NOT persist `disconnectWs` or `timers` (runtime-only).

**Refactored startMission():**
```typescript
async startMission(id: string, prompt: string): Promise<void> {
  const data = this.missions.get(id);
  if (!data) throw new Error(`Mission ${id} not found`);

  data.prompt = prompt;
  data.status = "running";
  data.startedAt = Date.now();

  // 1. Create container
  const config: SandboxConfig = {
    sessionId: id,
    workspaceDir: `/workspace/${id}`,
    // ... defaults from schema
  };
  const containerInfo = await this.containerManager.createSessionContainer(config, this.proxyPort);
  data.containerId = containerInfo.containerId;
  data.hostPort = containerInfo.hostPort;
  data.authToken = containerInfo.authToken;
  data.workspace = config.workspaceDir;

  // 2. Wait for control server ready
  await this.containerManager.waitForReady(containerInfo.hostPort, containerInfo.authToken);

  // 3. Start session timers
  data.timers = startSessionTimers(id, (sid, reason) => this.handleTimeout(sid, reason));

  // 4. Connect to event stream
  data.disconnectWs = this.containerClient.connectEvents(
    containerInfo.hostPort,
    containerInfo.authToken,
    (rawData) => {
      // Reset activity timer on each message
      data.timers?.resetActivity();

      if (rawData.type === "session_completed") {
        this.handleSessionComplete(id, rawData);
      } else if (rawData.type === "session_error") {
        this.handleSessionError(id, rawData);
      } else {
        // SDK envelope — relay to existing pipeline
        const envelope = rawData as SDKEnvelope;
        data.sdkMessages.push(envelope);
        this.notifySDK(id, envelope);
        this.saveMission(id);
      }
    },
    (err) => this.handleSessionError(id, { error: err.message }),
  );

  // 5. Send prompt to container
  await this.containerClient.startSession(containerInfo.hostPort, containerInfo.authToken, {
    prompt,
    sessionId: id,
  });

  this.saveMission(id, true);
}
```

**New helper methods:**
- `handleSessionComplete(id, data)`: Clear timers, copy JSONL from container, copy workspace files, destroy container, set status "completed", save snapshot.
- `handleSessionError(id, data)`: Same cleanup but set status "failed".
- `handleTimeout(id, reason)`: Log reason, abort session via ContainerClient, trigger cleanup.
- `async cleanup(id)`: Disconnect WS, clear timers, copy files, destroy container.

**Changes to MissionManager.create():**
- Instantiate `ContainerManager` and `ContainerClient`
- Call `containerManager.sweepOrphans()` on startup
- Accept `proxyPort` parameter (from credential proxy -- Plan 03)
- Default `proxyPort` to an env var `CREDENTIAL_PROXY_PORT` (default 9100) so Plan 02 can be implemented before Plan 03 exists

**Changes to workspace access:**
- `listWorkspaceFiles()` and `readWorkspaceFile()` -- for RUNNING sessions, proxy through `ContainerClient` (GET /files, GET /files/read) to the container. For COMPLETED sessions, read from host storage (SessionStore saved workspace).
- Keep the existing `walkDir()` for completed session workspace reads.

**Preserve backward compatibility:**
- Keep the `engine` field in MissionData (set to null always). This avoids breaking any code that checks `data.engine`.
- The existing `subscribe()`, `subscribeSDK()`, `notifyEvent()`, `notifySDK()` methods remain unchanged. Frontend receives the same events.
- `getMission()`, `listMissions()`, `getEvents()`, `getSDKMessages()`, `getChat()` remain unchanged.

**CRITICAL: Do not change `packages/backend/src/ws/handler.ts` or any frontend code.** The existing WS relay pipeline already works with SDKEnvelope objects. The only change is where those envelopes come from (container WS instead of direct MissionEngine callbacks).
