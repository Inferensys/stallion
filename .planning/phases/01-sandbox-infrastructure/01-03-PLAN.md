Build the credential proxy (HTTP reverse proxy injecting API keys), cost budget monitoring, and network isolation (iptables blocking direct API access). These are the security and budget enforcement components of the sandbox.

Purpose: The credential proxy is the core security mechanism -- CC inside the container sets `ANTHROPIC_BASE_URL` to the proxy, sends requests with a placeholder key, and the proxy injects the real key. This means raw API keys never enter the container. The cost monitor tracks `total_cost_usd` from SDK result messages and enforces budget caps. Network isolation blocks containers from bypassing the proxy.

```typescript
import http from "node:http";
import httpProxy from "http-proxy";

export class CredentialProxy {
  private proxy: httpProxy;
  private server: http.Server;
  private sessionKeys = new Map<string, string>();
  private port: number;
  private isAzureFondry: boolean;
  private targetUrl: string;

  constructor() {
    this.isAzureFondry = process.env.CLAUDE_CODE_USE_FOUNDRY === "1";

    if (this.isAzureFondry) {
      const resource = process.env.ANTHROPIC_FOUNDRY_RESOURCE;
      // Azure Foundry endpoint format: https://<resource>.services.ai.azure.com/api/...
      // The SDK constructs the full URL; we just need to forward to the Azure endpoint
      // For now, target is determined per-request from the incoming Host/URL
      this.targetUrl = `https://${resource}.services.ai.azure.com`;
    } else {
      this.targetUrl = "https://api.anthropic.com";
    }

    this.proxy = httpProxy.createProxyServer({
      target: this.targetUrl,
      changeOrigin: true,
      secure: true,
    });

    this.proxy.on("proxyReq", (proxyReq, req) => {
      const sessionId = req.headers["x-stallion-session"] as string;
      const realKey = sessionId ? this.sessionKeys.get(sessionId) : undefined;

      if (!realKey) {
        proxyReq.destroy(new Error("No API key for session"));
        return;
      }

      if (this.isAzureFondry) {
        proxyReq.setHeader("api-key", realKey);
      } else {
        proxyReq.setHeader("x-api-key", realKey);
      }
      // Remove placeholder keys the container might have sent
      proxyReq.removeHeader("anthropic-api-key");
    });

    this.proxy.on("error", (err, _req, res) => {
      if (res && "writeHead" in res) {
        (res as http.ServerResponse).writeHead(502);
        (res as http.ServerResponse).end(JSON.stringify({ error: err.message }));
      }
    });

    this.server = http.createServer((req, res) => {
      const sessionId = req.headers["x-stallion-session"] as string;
      if (!sessionId || !this.sessionKeys.has(sessionId)) {
        res.writeHead(403, { "Content-Type": "application/json" });
        res.end(JSON.stringify({ error: "Invalid or missing session" }));
        return;
      }
      this.proxy.web(req, res, {});
    });
  }

  async start(port: number): Promise<void> { ... }
  stop(): Promise<void> { ... }
  registerSession(sessionId: string, apiKey: string): void { ... }
  unregisterSession(sessionId: string): void { ... }
  getPort(): number { ... }
}
```

**CRITICAL -- Azure Foundry compatibility note:** The research flagged this as a LOW confidence area (open question #1). The SDK sends requests to the Azure Foundry endpoint using the `api-key` header (Azure convention). The `ANTHROPIC_BASE_URL` env var inside the container should point to our proxy. The proxy then forwards to the actual Azure endpoint. The proxy needs to detect the mode and inject the right header format. If during implementation the Azure header format turns out to be different, add a comment documenting the actual format discovered and adjust accordingly.

**CRITICAL -- x-stallion-session header injection:** The control server inside the container must add `x-stallion-session: <sessionId>` to all outbound requests to the proxy. This is how the proxy identifies which session is making the request. This header is added by configuring the SDK's environment inside the container. Since the SDK uses `ANTHROPIC_BASE_URL` and makes standard HTTP requests, the control server can set a custom environment variable that the SDK includes. HOWEVER, the SDK may not support custom headers on API requests. Alternative approach: use the proxy port itself as the session identifier (each session gets a unique proxy port, or the proxy inspects the source IP/port). Simpler approach for MVP: since each container has its own `ANTHROPIC_BASE_URL` pointing to the same proxy but with a session-specific path prefix, e.g., `http://host.docker.internal:9100/session/<sessionId>/`. The proxy strips the prefix and uses it for key lookup.

ACTUALLY, the simplest approach: inject the API key into the container as `ANTHROPIC_API_KEY=session-<sessionId>` (a recognizable placeholder, NOT the real key). The proxy extracts the session ID from the `x-api-key` or `api-key` header value (it starts with "session-"), looks up the real key, and replaces it. This way no custom headers needed -- the proxy just inspects the standard auth header the SDK already sends.

Implement the "placeholder key contains session ID" approach:
- Container env: `ANTHROPIC_API_KEY=session-<sessionId>`
- Proxy reads `x-api-key` (or `api-key`) from incoming request
- If value starts with `session-`: extract session ID, look up real key, replace header
- If value does NOT start with `session-`: reject (403)

**Create `packages/backend/src/sandbox/cost-monitor.ts`:**

```typescript
export class CostMonitor {
  private sessionCosts = new Map<string, number>();

  recordCost(sessionId: string, costUsd: number): void {
    const current = this.sessionCosts.get(sessionId) ?? 0;
    this.sessionCosts.set(sessionId, current + costUsd);
  }

  getCost(sessionId: string): number {
    return this.sessionCosts.get(sessionId) ?? 0;
  }

  checkBudget(sessionId: string, budgetUsd: number): { exceeded: boolean; total: number; budget: number } {
    const total = this.getCost(sessionId);
    return { exceeded: total >= budgetUsd, total, budget: budgetUsd };
  }

  // Called by MissionManager when relaying SDK events
  processSDKMessage(sessionId: string, msg: unknown): void {
    if (typeof msg === "object" && msg !== null && "type" in msg) {
      const m = msg as { type: string; total_cost_usd?: number };
      if (m.type === "result" && typeof m.total_cost_usd === "number") {
        // result message reports cumulative cost, not incremental
        // So we SET to this value, not add (avoids double-counting)
        this.sessionCosts.set(sessionId, m.total_cost_usd);
      }
    }
  }

  reset(sessionId: string): void {
    this.sessionCosts.delete(sessionId);
  }
}
```

IMPORTANT per research Pitfall 7: The SDK only reports `total_cost_usd` in `result` messages at `query()` completion. This means cost enforcement is POST-HOC -- we cannot abort mid-session based on cost. The budget check is used to:
1. WARN the user before starting a new session if they are approaching their budget
2. PREVENT starting a new session if budget is exhausted
3. Record actual cost after each query() for billing/display

For Phase 1, this is acceptable. Real-time cost enforcement would require a different SDK mechanism (e.g., `maxTurns` as a proxy, which is Claude's discretion per CONTEXT.md).

```typescript
import Docker from "dockerode";
import dns from "node:dns/promises";

export async function applyNetworkIsolation(container: Docker.Container): Promise<void> {
  // Resolve api.anthropic.com to current IPs
  const anthropicIps = await resolveHostIps("api.anthropic.com");
  // Also block claude.ai to prevent alternate access
  const claudeIps = await resolveHostIps("claude.ai");
  const allIps = [...new Set([...anthropicIps, ...claudeIps])];

  for (const ip of allIps) {
    await runExec(container, [
      "iptables", "-A", "OUTPUT",
      "-d", ip,
      "-j", "REJECT",
      "--reject-with", "tcp-reset",
    ]);
  }
  console.log(`Network isolation applied: blocked ${allIps.length} IPs for container ${container.id}`);
}

async function resolveHostIps(hostname: string): Promise<string[]> {
  try {
    const results = await dns.resolve4(hostname);
    return results;
  } catch {
    console.warn(`Could not resolve ${hostname} — skipping IP block`);
    return [];
  }
}

async function runExec(container: Docker.Container, cmd: string[]): Promise<void> {
  const exec = await container.exec({
    Cmd: cmd,
    AttachStdout: true,
    AttachStderr: true,
  });
  const stream = await exec.start({ hijack: false, stdin: false });
  await new Promise<void>((resolve) => stream.on("end", resolve));
}
```

NOTE per research Pitfall 4: iptables by IP is imperfect for CDN-hosted domains. IPs can change. This is acceptable for Phase 1 (MVP/dev). For production, the recommendation is a Docker `internal` network with explicit proxy routing. Add a comment documenting this limitation.
